🔐 Configurar Microsoft Entra ID en DeskEli

Guía para el equipo de IT — App Registration en Azure

Objetivo

Permitir que los usuarios de PatPrimo (o cualquier empresa) inicien sesión en DeskEli con sus credenciales de Microsoft 365 (SSO). Requiere registrar DeskEli como una App Registration en Azure y capturar 3 datos en el panel admin de DeskEli.

Ambiente Servidor: https://deskeli.eliotproyectos.tech
Redirect URI que se registra en Azure: https://deskeli.eliotproyectos.tech/auth/microsoft/callback

Paso a paso en Azure Portal

1Entrar a Azure Portal

Ir a https://portal.azure.com con una cuenta que tenga rol Application Administrator o Global Administrator del tenant.

2Registrar la aplicación

Buscar "Microsoft Entra ID" → App registrations → botón + New registration.

Configurar:

CampoValor
NameDeskEli - Sistema de Tickets
Supported account typesAccounts in this organizational directory only (single tenant)
Redirect URIPlatform: Web · URI: https://deskeli.eliotproyectos.tech/auth/microsoft/callback

Click Register.

3Copiar Tenant ID y Client ID

En la página Overview de la App recién creada, copiar los 2 GUIDs:

DatoVa a DeskEli como
Application (client) IDClient ID
Directory (tenant) IDTenant ID
4Generar Client Secret

En el menú lateral izquierdo: Certificates & secrets → tab Client secrets → + New client secret.

  1. Description: DeskEli auth 2026
  2. Expires: 24 months (recomendado)
  3. Click Add
⚠️ CRÍTICO Copiá el Value del secret AHORA mismo. Solo se muestra una vez. Después, aunque abras el panel, verás solo ••••••. Si lo perdés, hay que generar otro.
5Configurar permisos (API Permissions)

Menú lateral: API permissions → + Add a permission → Microsoft Graph → Delegated permissions.

Marcar y agregar estos 4:

PermisoDescripción
openidLogin básico
profileNombre completo
emailCorreo del usuario
User.ReadLeer perfil del usuario logueado

Después, click en el botón azul "Grant admin consent for [organización]" arriba. Requiere admin de Azure.

6(Opcional) Restringir por grupo

Si querés que solo ciertos usuarios del tenant puedan entrar a DeskEli:

  1. En Overview de la App → click en el nombre azul a la derecha de "Managed application in local directory"
  2. Menu lateral: Properties → Assignment required = Yes → Save
  3. Menu lateral: Users and groups → + Add user/group → agregar el grupo de AD/Entra que puede usar DeskEli

Configurar en DeskEli

Una vez que IT te pase los 3 datos:

  1. Entrar a DeskEli como admin: https://deskeli.eliotproyectos.tech/admin/config
  2. Tab 🔐 Microsoft Entra ID
  3. Click ⚙️ Configurar en Manufacturas Eliot (o la empresa correspondiente)
  4. Llenar:
    • Tenant ID: el GUID del tenant
    • Client ID: el Application ID
    • Client Secret: el value del secret
    • ☑️ Habilitar login con Microsoft
  5. Click 💾 Guardar
✓ Listo En la pantalla de login aparece ahora el botón "Iniciar sesión con Microsoft". Los usuarios del tenant pueden entrar sin contraseña local.

Cómo funciona el flujo

1. Usuario en /login: click en "Iniciar con Microsoft"
   ↓
2. DeskEli redirige a login.microsoftonline.com
   ↓
3. Usuario autentica (con su clave de M365, MFA si aplica)
   ↓
4. Microsoft redirige a /auth/microsoft/callback con code
   ↓
5. DeskEli intercambia code por token, obtiene:
   - Object ID (identificador único)
   - Email / userPrincipalName
   - Nombre completo
   ↓
6a. Si existe usuario en DeskEli con ese Object ID → login OK
6b. Si existe con ese email → linkeo el Object ID y login OK
6c. Si NO existe → se auto-provisiona con rol "employee" por default
   ↓
7. Sesión creada → redirect a dashboard según rol

Preguntas frecuentes

¿Se puede usar la misma App Registration para las 3 empresas?

Sí, si las 3 usan el mismo tenant. Configurás los mismos Tenant ID + Client ID + Secret en las 3 empresas del panel DeskEli.

¿Qué pasa si el usuario cambia su contraseña de M365?

Nada. DeskEli no guarda contraseñas Microsoft — solo el Object ID. Al siguiente login, Microsoft valida y todo sigue funcionando.

¿Puedo revocar el acceso de un usuario?

Sí, hay 3 formas:

  1. En Azure: quitar el usuario del grupo con acceso a DeskEli (paso 6)
  2. En DeskEli: desactivar el usuario (is_active=false)
  3. Rotar Client Secret (echa a TODOS): generar uno nuevo en Azure, actualizarlo en DeskEli. Todos los tokens activos siguen funcionando hasta que expiren, pero nuevos logins requieren el secret actualizado.

¿Los usuarios que ya existían con contraseña local pueden seguir entrando?

Sí, ambos métodos coexisten. Un usuario puede entrar con Microsoft O con email/contraseña local. En el primer login con Microsoft, la cuenta se linkea automáticamente (por email) al Object ID de Entra.

¿Y si un usuario nuevo del tenant quiere entrar?

Se le crea la cuenta automáticamente en DeskEli con rol employee. Después el admin puede promoverlo a técnico o admin desde el panel de Equipo.

⚠️ Seguridad del Client Secret El secret se guarda cifrado en la BD (con DB_ENCRYPTION_KEY) y auditado. Aún así, tratátalo como una contraseña de servicio: gestor de secretos, rotación cada 12-24 meses.
